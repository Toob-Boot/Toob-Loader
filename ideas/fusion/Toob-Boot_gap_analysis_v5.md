# Gap-Analysis V5: Toob-Boot Architektur-Synthese (Die "Mission-Critical" Edge-Cases)

Die Version 4 und das zugehörige `capability_analysis_v2` Update haben Toob-Boot endgültig aus der Kategorie der "Guten Bastel-Projekte" in die Liga von sicherheitskritischen Automotive- und Aerospace-Frameworks katapultiert (TMR, Hardware-Reset-Reasons, Dual-Bank S1).

Die V5-Prüfung testet das Konstrukt nun auf Hardware-Hacking-Ebene (Timing/Glitching) und tiefgreifenden Multi-Core-Abhängigkeiten. 

Hier sind die vier absolut tiefsten, verbleibenden Systems-Engineering Gaps:

### 1. JTAG/SWD Sperre via `startup.c` ist ein Race-Condition-Spielzeug (Security / Hardware Exploit)
**Problem:** Abschnitt 1B schreibt vor, dass die `startup.c` die JTAG/SWD-Ports sperren muss, *bevor* Code ausgeführt wird (`security.lock_debug_port = true`). Das ist ein Widerspruch in sich. Die `startup.c` *ist* Codeausführung. Zwischen dem Power-On-Reset (POR) der Hardware und dem Ausführen der allerersten Instruktion in `startup.c` vergehen teils Millisekunden (während Boot-ROM Parameter lädt oder PlLa/Clocks einschwingen). Ein Hardware-Hacker schließt einen physikalischen SWD-Debugger an und feuert in einer Endlosschleife "HALT"-Commands via JTAG ein. Der Core stoppt im Reset-Handler **vor** der `startup.c` Sperre. Der Angreifer kann den RAM auslesen/manipulieren oder direkt in Funktionen springen.
**Mitigation:** Software-basierte JTAG-Sperren sind unzureichend. Die Dokumentation muss zwingend fordern, dass JTAG/SWD über **Hardware-Fuses (eFuses / Flash Option Bytes)** deaktiviert werden muss, welche vom internen Silizium-Boot-ROM evaluiert werden, bevor der Reset-Vector auch nur angerührt wird. Die `startup.c`-Sperre darf nur als "Defense-in-Depth" Fallback dienen.

### 2. Multi-Core Update "Amnesie" & Kompatibilitäts-Crash (Logic / IPC)
**Problem:** In Abschnitt 1B ist nun Multi-Core verankert (`[images.app]`, `[images.netcore]`). Wenn ein Update beide Images beinhaltet und das Gerät das `app` Image erfolgreich flasht, das `netcore` Image beim Flashen aber einen Flash-Defekt erfährt (oder Batterie stirbt ohne Recovery-Möglichkeit), führt Toob-Boot für `netcore` einen Rollback aus. Dann bootet die *neue* Main-App mit dem *alten* Netcore. Wenn beide über Shared Memory/IPC kommunizieren und sich C-Structs geändert haben, sprengen sie sich beim Boot gegenseitig in die Luft (Mismatched ABI).
**Mitigation:** Multi-Image Architekturen benötigen zwingend das Konzept einer **Atomic Update Group ("Lock-Step")**. Das gesamte SUIT-Manifest darf nur einen einzigen, globalen `TXN_COMMIT` für die ganze Update-Gruppe zulassen. Wenn ein einziges Sub-Image in der Verifikationsphase verfehlt, MÜSSEN alle anderen, bereits in die Slots geschriebenen Images der Gruppe ebenfalls einem Rollback unterzogen werden.

### 3. Stage 0 Selektions-Dilemma bei S1 Dual-Bank (Availability / Routing)
**Problem:** Schicht 3 definiert "S1a / S1b" (Dual-Bank für Stage 1). Stage 0 (Immutable) muss nun entscheiden, welches Stage 1 es bootet. Wie macht Stage 0 das? Wenn es einfach das validiert, welches an Adresse A liegt und bei Fehler Adresse B nimmt, wie iteriert man Upgrades? Angenommen S1a ist v1, S1b ist das neue v2. Stage 0 schaut zuerst nach S1a, es ist valide (weil alte Version noch existent), also bootet Stage 0 immer die alte v1, es sei denn, man löscht absichtlich S1a. Aber beim Update-Vorgang löscht man die alte Version nie, bevor die neue bestätigt ist! Stage 0 braucht eine Information.
**Mitigation:** Das Konzept muss exakt vorgeben, dass Stage 0 einen absolut simplen, stromausfallsicheren S1-Routing-Pointer auswertet. Am Ende der eFuse-Tabelle oder in einem rudimentären "Boot_Pointer" Flash-Sector wird ein einziges Byte hinterlegt (z.B. `0xA = Boot S1a, 0xB = Boot S1b`). Wenn Stage 1 sich selbst updatet (auf den Offline-Slot), verifiziert es sich intern und dreht erst dann dieses atomare State-Byte um. Ohne diesen Selector läuft Stage 0 bei Dual-Bank blind ins Leere.

### 4. Stage 1 Self-Update "ABI Break" (Maintainability / Dev-Experience)
**Problem:** Wenn Stage 1 (alt) das Stage 1 (neu) aktualisiert, hängen beide Versionen von demselben physischen Flash-Journal (WAL) ab. Hat der Maintainer in Stage 1 v2 eine geniale Idee und verändert die `struct wal_entry` (z.B. von 128 Bytes auf 132 Bytes), passiert folgendes: S1 (alt) bootet S1 (neu). S1 (neu) liest das WAL, parst es mit dem 132-Byte Offset, bekommt Garbage-Daten, stürzt als "Corrupted State" ab und löst im besten Fall Rollbacks oder im schlimmsten Fall Bricks aus.
**Mitigation:** Das Dokument (DX / Maintainability) muss den Entwicklern klarmachen: **Die Struktur-Layouts (ABI) des WAL und des `Current_Primary_Slot` sind heilige Verträge**. Es muss einen "ABI-Version" Identifier geben. Bevor ein neues Stage 1 das alte WAL verarbeitet, MUSS es eine strukturierte Migration-Routine besitzen, die das WAL-Speicherformat von v1 auf v2 portiert. Das "Data-Format" des Bootloaders darf niemals stillschweigend mutieren.
