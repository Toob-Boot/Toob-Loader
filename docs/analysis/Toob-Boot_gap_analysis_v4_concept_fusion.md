# Toob-Boot Architektur: Gap-Analyse V4 (Production Edge-Cases)

Basierend auf den Prinzipien einer absolut fehlertoleranten Industrie-Architektur ('Power of 10', P10) und den harten Hardware-Realitäten moderner SoCs, wurden in der vorliegenden Spezifikation `docs/concept_fusion.md` die folgenden 10 kritischen logischen Gaps und architektonischen Blind Spots identifiziert:

---

## 1. Kaltstart-Abhängigkeit verhindert echte Update-Archivierung (Logic & Edge-Case)
**Problem:** Abschnitt 1 schreibt vor: *"Das OS muss exakt diese Nonce als Confirm-Flag ins NV/RTC-RAM zurückschreiben... Die finale Archivierung erfolgt passiv beim nächsten natürlichen Kaltstart."*
Wenn ein IoT-System oder Smart Meter an einer starken Stromquelle jahrelang fehlerfrei durcharbeitet (kein Reboot), verbleibt das System über Jahre im Zustand "Tentativ/Probe". Fällt nach diesen Jahren der Strom aus (echter Kaltstart), ist das RTC-RAM physikalisch gelöscht. Stage 1 fährt hoch, sieht ein leeres Confirm-Flag und führt einen desaströsen Rollback auf die veraltete Firmware der vorherigen Dekade durch.
**Die BESTE Mitigation:** Keine exklusive Abhängigkeit vom Hardware-Reboot. Es muss eine asynchrone **"S1_Commit_Mailbox"** (via HAL oder ein dediziertes IPC-Speicherfeld) implementiert werden. Das OS muss Stage 1 zur Laufzeit aktiv signalisieren können: "Ich lebe stabil, verankere das Image jetzt!". S1 vollendet den Prozess und eliminiert den Tentativ-Status persistent, auch ohne sofortigen Reboot.

## 2. Dynamische Speicherallokation zerstört SRAM-Determinismus (Security & Efficiency)
**Problem:** Laufzeit-Garantien werden kompromittiert, da Schicht 2 ("OS Shim") `malloc/free` erlaubt, *"falls die Crypto-Lib es fordert"*. Der Build-Automatisierungs-Report verspricht eine deterministische `Peak_SRAM`-Prüfung, aber eine beliebige Heap-Allokation (`malloc`) macht dies zur Buildzeit komplett unverifizierbar. Es kann durch fehlerhafte Pakete/SUIT-Ketten zur Laufzeit zu Heap-Fragmentation oder OOM-Fehlern kommen, die Stage 1 unkalkulierbar abstürzen lassen.
**Die BESTE Mitigation:** Striktes Verbot von `malloc/free` (Heap) im kompletten Stage 1! Krypto-Bibliotheken (wie Ed25519-donna oder Monocypher) unterstützen ausnahmslos static memory allocation. Der Compiler MUSS einen dedizierten, festen SRAM-Buffer (`uint8_t crypto_arena[1024]`) definieren, auf dem gearbeitet wird. Nur so kann der Peak-Mem hart vom Linker bewiesen werden.

## 3. Feedback-Blindflug des OS nach fehlgeschlagenem Delta-Update (Usability/Dev Experience)
**Problem:** In Sektion 3 heißt es: *"Weist Stage 1 das Delta wegen solider Bit-Rot physikalischer Alterung ab, loggt das System einen Fallback-to-Full Intent im Journal."* Ohne ein dediziertes Recovery-OS startet danach einfach der alte Slot A normal. Da das Konzept keine Status-Rückmeldung an das OS vorsieht, weiß der OTA-Agent des OS nicht, warum sein Update vorhin beim Booten fehlschlug. Das OS fordert beim nächsten Backend-Poll einfach das exakt gleiche, nutzlose Delta-Paket nochmal an und gerät in eine unendliche Schleife.
**Die BESTE Mitigation:** Schicht 4b (`Diagnostics`) MUSS eine `boot_diag_get_last_error()` API / Shared-Memory-Struktur im `.noinit` Areal bereitstellen. Das Feature-OS muss den Grund (z.B. `ERR_DELTA_BASE_MISMATCH`) beim Start auslesen können, um beim Cloud-Backend proaktiv ein `/full` Image anstelle eines `/delta` Pakets anzufordern.

## 4. Replay-Attacken durch "Highest-Timestamp" Verlust im zirkulären WAL (Security)
**Problem:** Um Serial-Recovery (Schicht 4a) gegen Replay-Attacken lokaler Angreifer abzusichern, wird ein `Highest_Timestamp` Konzept gefahren. Da dieser Counter aber laut Konzept *„im WAL-NVRAM gespeichert“* wird, existiert eine massive Lücke: Das WAL ist zirkulär und verliert bei Formatierungen oder Rollbacks (nach maximalem Verschleiß) seinen alten Inhalt. Der Zähler fällt auf "0" und jahrealte, einst abgehörte 2FA-Recovery Tokens der Werkstechniker sind plötzlich wieder vollständig gültig!
**Die BESTE Mitigation:** Absolute, monotone Zeitstempel dürfen nicht im flüchtigen WAL-Layer verbleiben. Sie MÜSSEN in isolierten, verschleißfesten Persistenz-Schichten gesichert werden (Data-EEPROM mit eigenem Endurance-Rating oder 1-Way Option-Bytes, so nah an echter eFuse-Logik wie das Silicon es zulässt).

## 5. Crash-Risiko durch Watchdog-Prescaler Rundungsfehler (Hardware Reality)
**Problem:** Sektion 1B berechnet dynamisch: `längste_Einzeloperation + 2x_Marge`. Hardware-Watchdogs (z. B. IWDG auf ARM oder TIMG im Xtensa) greifen jedoch nur starre Prescaler-Teiler (z.B. Timeout exakt alle /4 = 2.0s, /8 = 4.1s). Ermittelt der Compiler beispielsweise 3.5 Sekunden als absolut nötigen Wert und drückt diesen in die HAL, aber die Hardware beherrscht physikalisch als Nächstes nur 2.0s oder 4.1s, entstehen Abrundungsrisiken. Rundet das System ab, resettet der Target-Chip bei extremen Sektor-Erase Operationen viel zu früh.
**Die BESTE Mitigation:** Das Tooling braucht `WDT_Quantum` Maps in den Vendor-Plugins (`device.toml`). Der Preflight-Report darf nicht das logische "mathematische Maximum" bestimmen, sondern MUSS den errechneten T-Max strikt auf die **nächstmögliche, physikalisch garantierte Hardware-Treppenstufe** der jeweiligen MCU aufrunden.

## 6. Cloud-Lockout "72h TTL" führt physisch gelagerte Hardware ins Verderben (Usability)
**Problem:** Die Regel *"Ist die Cloud >72h offline ... löscht das OS das Confirm-Flag"* ist fatal fehleranfällig für gelagerte Hardware. Wenn ein voll funktionales Firmware-Update auf ein Gerät installiert wird, und der Kunde das Gerät danach versehentlich für eine Woche in eine Box ohne Empfang stellt oder komplett vom Stromnetz trennt, hat das Gerät logisch beim nächsten Aufwachen "verspätet", da über die RTC verifiziert wurde, dass >72h vergangen sind. Es bricht grundlos das intakte Update ab!
**Die BESTE Mitigation:** Der Network-Watchdog (Fallback) des OS darf sich absolut nicht nach reiner Wanduhrzeit (Echtzeit/TTL) richten. Die Metric MUSS zwingend **„Aktiv akkumulierte Netz-Suchzeit“** (z. B. "Nach summierten 10 vollen Betriebsstunden vergeblicher TCP-Paket-Repeats") als Anker nutzen, um gelagerte Systeme immun gegen sinnlose Timeout-Rollbacks zu machen.

## 7. Fehlender Safe-Lockdown State bei absolutiertem Flash-Verschleiß (Maintainability)
**Problem:** Es existiert zwar ein Indikator *"mahnt Fleet-Manager bei 80.000 Zyklen"*, doch das Verhalten nach Erreichen der absoluten Zerstörungsgrenze (z. B. 100.000 Erase-Zyklen) ist unbestimmt. Lässt Stage 1 blindwütig weitere Updates zu, wird das Halbleitermaterial kollabieren, Erases werden als Exception fehlschlagen und das Gerät endet unkontrolliert in einem harten Brick oder endlosen Crash-Loop.
**Die BESTE Mitigation:** Einführung eines harten **`End-Of-Life (EOL) Survival Modes`**. Sobald der absolute Maximalzyklus des Flashs laut HAL/Zählerstand erreicht ist, MUSS Stage 1 weitere Schreib/Erase-Vorgänge hard blockieren (außer im WAL), den Bootloader-Zustand auf `STATE_READ_ONLY` setzen und dies dem OS melden. Die Hardware rettet sich so als funktionsfähiger Sensor vor ihrem eigenen hardwareseitigen Exitus.

## 8. Atomic Multi-Core Grouping kollidiert mit isolierten Bussen (Architektur)
**Problem:** Die Spezifikation fordert das absolute Lock-Step: *"Wenn ein Sub-Image bricht, rollen ALLE assoziierten Images zurück"*. Bei Systemen, in denen das Radio-Modul physikalisch völlig separiert auf einem isolierten Bus agiert, hat Stage 1 oftmals gar keinen PIN-Zugriff auf dessen Flash (oder nur, wenn das Main-OS den Bus initialisiert hat). Ein Rollback der fehlerhaften Main-App auf Version V1 hinterlässt den externen Sub-Core unwartbar auf Version V2. Totaler ABI Crash!
**Die BESTE Mitigation:** Definition einer asymmetrischen Rollback-Hierarchie (`Dependency Graph`) in der Architektur. Ist ein Sub-Core hardware-isoliert, MUSS der Sub-Core über seinen eigenen winzigen Boot-Agent ("Stage 1.5") gestartet werden. Bootet die Main-App (z.B. zurückgerollt auf V1), übermittelt sie ihre Version via IPC an den Sub-Core zur Laufzeit. Der Sub-Core Agent erkennt die Inkompatibilität manuell und initiiert seinen eigenen, lokalen Rollback aus seinem lokalen Backup-Sektor.

## 9. Wear-Zerstörung durch täglichen "Double-Flash-Confirm" (Hardware/Efficiency)
**Problem:** In `confirm_hal_t` steht: *"Ob schnelles RTC-RAM... oder ein wear-leveled Flash-Sector für MCU-Kaltstarts ohne Knopfzelle"*. Wenn eine MCU kein gestütztes RTC-RAM besitzt und das System ein klassisches IoT Edge-Device ist, muss sich das Gerät *jeden* Tag nach dem Deep-Sleep neu über den Bootloader initialisieren. Das OS würde bei diesem Setup bei *jedem einzelnen Aufwachen* im Kaltstart einen Confirm-Flag Block auf dem Flash löschen und neu schreiben, nur um S1 zu besänftigen! Das tötet den Flash enorm schnell.
**Die BESTE Mitigation:** Wenn die Plattform zur Confirm-Speicherung zwangsweise den Flash verwendet, muss der Bootloader dem OS bei Systemstart signalisieren, ob ein Confirm überhaupt aussteht (`bool is_update_pending_confirm`). Das OS verfeuert den Flash-Erase Zyklus somit exakt EIN EINZIGES MAL nach einem Flash-Prozess (und nicht 300x im Jahr). Das MUSS expliziter API-Contract der Architektur sein!

## 10. "Malleability by Truncation" der Chunks im WAL-Stream-Betrieb (Security/Logic)
**Problem:** In Sektion 3 wird eine *"Envelope-Wrap Validierung (Signatur über das gesamte Paktet)"* gefordert, da eine Signatur das Gesamtpaket schützt. In der Stream-Erläuterung steht jedoch: `1. Empfang (Chunk-weise) -> 2. Chunk-Hash -> 3. WAL-Eintrag -> Write`. Wenn S1 empfangene Pakete sofort flasht (WAL), während die 100%ige Überprüfung der Signatur noch aussteht, können Hacker manipulierte Daten schreiben (die Signatur würde am Ende fehlschlagen, aber die Chunks haben den Flash bereits verändert)! Dies bricht das Fundament der Sicherheit, denn Stage 1 hat unautorisierten Payload vor der letzten Signatur-Zusage verarbeitet.
**Die BESTE Mitigation:** SUIT-Manifeste MÜSSEN zwingend asymmetrisch organisiert sein: Der Header (`Manifest`) inkludiert den asymmetrischen Root-Hash. S1 MUSS sofort via Ed25519 die Signatur validieren. Sobald der Header mathematisch als "Authentisch" bewiesen ist, kennt S1 den sicheren Ed25519-Hashtree/RootHash. ERST JETZT dürfen Chunks gestreamt, pro-Chunk gegen den sicheren Hash geprüft und geflasht werden. Die Reihenfolge "Verify first, stream later" muss strikt als Gesetz definiert werden, nicht als "Check at EOF".
